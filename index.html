Best, Average & Worst Case Analysis
Best Case
O(n·W)
Same as worst case — the table always has (n+1)×(W+1) cells regardless of item values. Every cell is visited exactly once.
When: Any input — DP always fills the full table.
Average Case
O(n·W)
Still O(nW) — DP has no early exit. On average, roughly half the items will be included in the optimal set, but all cells are still computed.
When: Random items with mixed weights and values.
Worst Case
O(n·W)
When all items have weight=1 and W is large — every cell requires a max() comparison with no items being skipped due to weight overflow.

Why is O(2ⁿ) Brute Force so bad? ►

For n=20 items, brute force checks 220 = 1,048,576 subsets. For n=30, over 1 billion. DP with W=20 needs only 20×20 = 400 operations.

0/1 Knapsack is "pseudo-polynomial": O(nW) is efficient when W is small, but technically exponential in the bit-length of W.

💡 Pseudo-polynomial: O(nW) is practical when W is bounded, but not truly polynomial in the input size (bit-length of W).
💡 Worst case for DP occurs when all items fit and W is large — every cell requires a full comparison. Try the "Worst Case" preset to see maximum table fill.
💡 Space optimisation: Space can be reduced from O(nW) to O(W) using a single 1D array filled right-to-left, since row i only depends on row i−1.
⚠ Common mistake: Assuming O(nW) is polynomial time. It is only pseudo-polynomial — if W = 2n, DP is no better than brute force.
CLO-2: Sorting & Searching
Role of Sorting: Value/Weight Ratio Pre-Sort ►

Sorting items by v/w ratio descending is the Greedy approach for Fractional Knapsack. For 0/1 Knapsack, DP does not need this sort, but understanding it shows why greedy fails.

In the traceback phase, the algorithm performs a linear search (O(n)) through the DP table rows to find which items were selected — starting at K[n][W] and walking upward.

💡 Sorting role: v/w ratio sort = greedy heuristic order. DP fills the table correctly regardless of item order.
💡 Traceback search: After filling the table, compare K[i][w] vs K[i−1][w] for each row. If they differ, item i was included. This is O(n) sequential search.
💡 Fractional vs 0/1: Greedy (sort by v/w then take fractions) solves Fractional Knapsack optimally in O(n log n). For 0/1 Knapsack, greedy is a heuristic that can fail — see the counterexample below.
⚠ Common mistake: Applying greedy sort to 0/1 Knapsack and expecting an optimal result. The inability to take fractions breaks greedy optimality.
⚠ Common mistake: Skipping traceback and only reporting K[n][W]. The value is correct but you cannot determine which items to pack without the O(n) traceback search.

CLO-3: Design Paradigm - Dynamic Programming
Paradigm: Dynamic Programming
Recurrence: K[i][w] = max(K[i-1][w], vᵢ + K[i-1][w-wᵢ])
How DP Works: Overlapping Subproblems & Optimal Substructure ►

Optimal Substructure: The optimal solution to K[i][w] depends on optimal solutions to K[i−1][w] (exclude) and K[i−1][w−wᵢ] (include). You can build the global optimum from smaller optimal subproblems.

Overlapping Subproblems: Many cells K[i][w] are reused across different decision branches. DP stores results in a table (memoization) instead of recomputing them — this is what makes it efficient.

💡 DP = Recursion + Memoization. Fill small subproblems first (row 0, then row 1…), building bottom-up to the full answer K[n][W].
💡 Why not Divide & Conquer? D&C subproblems are independent (e.g. merge sort splits into non-overlapping halves). Knapsack subproblems overlap — that overlap is exactly what DP exploits.
💡 Why not Greedy? Greedy makes a locally optimal choice (highest v/w) at each step without reconsidering. Knapsack requires global optimisation across all combinations — greedy cannot guarantee this (see counterexample below).
💡 Bottom-up vs Top-down: This visualiser uses bottom-up DP (fill the table iteratively). Top-down DP uses recursion with a memoisation cache — same result, different execution order.
⚠ Common mistake: Confusing DP with Divide & Conquer. D&C splits into independent subproblems; DP is for overlapping subproblems.
⚠ Common mistake: Thinking the recurrence K[i][w] = max(include, exclude) only works for 2 choices. It generalises: Unbounded Knapsack changes K[i-1] to K[i] in the include branch, allowing reuse of the same item.
Greedy Counterexample: When Greedy Fails
Greedy picks highest v/w first but may leave capacity wasted. DP always finds true optimum.

Algorithm Overview
What is the 0/1 Knapsack Problem? ►

Given n items each with weight wᵢ and value vᵢ, and a knapsack of capacity W: find the subset maximising total value without exceeding W. Each item is taken (1) or not (0).

When to use it

Resource allocation: project budgeting, cargo loading, portfolio optimisation, bin packing.

How it works

Build a 2D table K[i][w] = best value using items 1..i with capacity w. Fill row by row. Answer is K[n][W].

K[i][w] = max(K[i-1][w], vᵢ + K[i-1][w-wᵢ]) if wᵢ ≤ w, else K[i-1][w]
Worked Example - Step by Step ►

Items: i1(w=2,v=6), i2(w=2,v=10), i3(w=3,v=12) | W=5

Fill Table:

i=1: K[1][2..5]=6 (include i1, w=2)

i=2: K[2][2]=max(6,10)=10, K[2][4]=max(6,16)=16

i=3: K[3][5]=max(K[2][5], 12+K[2][2])=max(16,22)=22

Traceback from K[3][5]=22:

K[3][5]≠K[2][5] → i3 included, go to K[2][2].
K[2][2]≠K[1][2] → i2 included. Done.

Answer: {i2, i3}, value=22, weight=5

Key Concepts & Common Mistakes ►
0/1 constraint: Cannot take a fraction. Full weight and value or nothing.
Row independence: Row i depends only on row i-1. Can optimise space to O(W) with 1D array.
⚠ Using greedy (highest v/w) for 0/1 Knapsack fails. Greedy works for Fractional Knapsack only.
⚠ Forgetting traceback: K[n][W] gives the value but you need traceback to find which items.
⚠ Off-by-one: Row 0 and column 0 must be all zeros (base case).
Variants & Extensions ►

Unbounded Knapsack: Each item can be taken multiple times. Change to K[i][w] = max(K[i-1][w], vᵢ + K[i][w-wᵢ]).

Fractional Knapsack: Can take fractions. Solved greedily in O(n log n).

Space Optimisation: Use 1D array of size W+1, update right-to-left: O(W) space.

Subset sum is a special case of 0/1 Knapsack where vᵢ = wᵢ.
